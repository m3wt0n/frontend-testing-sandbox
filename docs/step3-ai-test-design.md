# ステップ3：AI を使ったテスト設計

## 概要

AI をテスト設計に活用する目的は「テストを全部書かせる」ことではない。
**人間が見落としがちなエッジケースの発見・テスト設計の網羅性チェック・定型コードの生成**を AI に任せ、
「何をテストすべきか」の判断は人間が行うのが正しい使い方。

---

## AI テスト設計の活用パターン全体像

```
┌─────────────────────────────────────────────────────┐
│ 1. テストケース設計の壁打ち                          │
│    仕様・コードを渡して「何をテストすべきか」を聞く   │
├─────────────────────────────────────────────────────┤
│ 2. エッジケースの洗い出し                            │
│    正常系を書いたあとに「他に何があるか」を聞く       │
├─────────────────────────────────────────────────────┤
│ 3. テストコードの生成                                │
│    設計済みのテストケースをコードに落とす             │
├─────────────────────────────────────────────────────┤
│ 4. 既存テストのレビュー・改善提案                    │
│    書いたテストの品質・抜け漏れを確認する             │
├─────────────────────────────────────────────────────┤
│ 5. Storybook ストーリー・play() の生成               │
│    コンポーネントのインタラクションシナリオを作る      │
└─────────────────────────────────────────────────────┘
```

## パターン1：テストケース設計の壁打ち

コードと仕様を渡して「何をテストすべきか」をリストアップしてもらう。
コードを書く前に使うと、テスト設計の抜け漏れを防げる。

### プロンプトテンプレート

```
以下の React コンポーネントに対して、テストケースを設計してください。

【コンポーネントのコード】
```tsx
// ここにコンポーネントのコードを貼る
```

【仕様・要件】
- （箇条書きで仕様を書く）

【お願いしたいこと】
- Testing Library の思想（ユーザー視点）でテストケースを列挙してください
- 正常系・異常系・エッジケースに分類してください
- テストコードはまだ書かなくてよいです。ケースの一覧だけください
```

### 実例：フォームコンポーネントの場合

**入力（プロンプト）：**

```
以下のログインフォームコンポーネントに対して、テストケースを設計してください。

【コンポーネントのコード】
```tsx
export const LoginForm = ({ onSubmit }: { onSubmit: (data: LoginData) => void }) => {
  const [email, setEmail] = useState('')
  const [password, setPassword] = useState('')
  const [errors, setErrors] = useState<Record<string, string>>({})

  const validate = () => {
    const newErrors: Record<string, string> = {}
    if (!email) newErrors.email = 'メールアドレスを入力してください'
    if (!email.includes('@')) newErrors.email = '正しいメールアドレスを入力してください'
    if (!password) newErrors.password = 'パスワードを入力してください'
    if (password.length < 8) newErrors.password = 'パスワードは8文字以上で入力してください'
    return newErrors
  }

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault()
    const newErrors = validate()
    if (Object.keys(newErrors).length > 0) {
      setErrors(newErrors)
      return
    }
    onSubmit({ email, password })
  }
  // ...略
}
```

【お願いしたいこと】
テストケースを正常系・異常系・エッジケースに分類して列挙してください。
```

**AI の出力例（Claude の回答）：**

```
【正常系】
- 有効なメールアドレスとパスワード（8文字以上）を入力して送信すると onSubmit が呼ばれる
- 送信後にフォームがリセットされる（仕様に依存）

【異常系 - バリデーション】
- メールアドレスが空のとき「メールアドレスを入力してください」が表示される
- メールアドレスに @ が含まれないとき「正しいメールアドレスを入力してください」が表示される
- パスワードが空のとき「パスワードを入力してください」が表示される
- パスワードが7文字以下のとき「パスワードは8文字以上で入力してください」が表示される
- 両方空のとき、両方のエラーが表示される
- バリデーションエラーがあるとき onSubmit が呼ばれない

【エッジケース】
- メールアドレスに @ が複数含まれる場合（例: a@@b.com）→ 現在の実装では通過してしまう
- パスワードがちょうど8文字の場合 → 境界値テスト
- パスワードが7文字の場合 → 境界値テスト
- メールアドレスが空白のみの場合（例: "   "）→ 現在の実装ではバリデーションを通過する可能性
- スペースを含むパスワードの場合
- 一度エラーが出たあと正しく入力し直して送信できるか

【アクセシビリティ観点】
- 各入力欄にラベルが正しく紐づいているか（getByLabelText で取得できるか）
- エラーメッセージが aria-live 領域か alert ロールで通知されるか
```

→ このリストを見て「@ が複数の場合は今回スコープ外」「空白のみのケースはバグなので修正してテストも追加」など、**人間が判断してからテストコードを書く**のが正しい流れ。

## パターン3：テストコードの生成

設計済みのテストケース一覧をコードに落とす。
**「何をテストするか」を先に決めてから** コード生成を頼むのがポイント。
ケース設計なしに「テストを書いて」と頼むと、表面的なテストしか生成されないことが多い。

### プロンプトテンプレート

```
以下のテストケースを、Vitest + @testing-library/react + userEvent で実装してください。

【テスト対象のコンポーネント】
```tsx
// コードを貼る
```

【実装してほしいテストケース】
1. 有効な入力で送信すると onSubmit が呼ばれる
2. メールアドレスが空のとき送信できずエラーが表示される
3. パスワードが7文字以下のとき送信できずエラーが表示される

【制約】
- getByRole を最優先に使ってください
- userEvent.setup() を使ってください
- describe / it のネスト構造を適切にしてください
- vi.fn() でモックを作ってください
```

### 生成されたコードは必ずレビューする

AI が生成したテストコードに多いアンチパターン：

```tsx
// アンチパターン1：getByTestId に頼りすぎる
screen.getByTestId('submit-button')    // テスト ID はなるべく使わない

// アンチパターン2：実装の詳細に依存する
expect(component.state.email).toBe('') // 内部状態を直接見ている

// アンチパターン3：fireEvent を使っている
fireEvent.click(button)                // userEvent を使うべき

// アンチパターン4：非同期を待っていない
expect(screen.getByText('成功')).toBeInTheDocument() // findBy か waitFor を使うべき
```

## GitHub Copilot を使う場合のコツ

Copilot は「次の1行」を補完するのが得意。テスト設計よりも**テストコードの実装補助**に向いている。

### 効果的な使い方

```tsx
// コメントで意図を書くと補完精度が上がる
// メールアドレスが空のとき送信できずエラーメッセージが表示されること
it('', () => {
  // ← ここで Copilot に補完させる
})
```

```tsx
// describe のブロックを先に書いておくと補完がブレにくい
describe('LoginForm', () => {
  describe('バリデーション', () => {
    it('メールアドレスが空のとき', async () => {
      // ← より具体的な補完が得られる
    })
  })
})
```

### Copilot Chat でのレビュー

エディタ上で選択してから Copilot Chat に聞く：

```
このテストコードを Testing Library のベストプラクティスに沿ってリファクタしてください。
getByTestId は使わず、getByRole や getByLabelText を優先してください。
```

## AI テスト設計のワークフロー（まとめ）

実際の開発サイクルに組み込む場合の推奨フロー：

```
1. コンポーネントの仕様・コードを用意する
        ↓
2. AI にテストケース一覧を出してもらう（パターン1）
        ↓
3. 人間がテストケースをレビューして取捨選択する
   ・スコープ外のケースを除く
   ・仕様バグを発見したら先に修正する
   ・優先度をつける
        ↓
4. AI にテストコードを生成してもらう（パターン3）
        ↓
5. 人間がコードをレビューしてアンチパターンを修正する
        ↓
6. AI にエッジケースの追加提案を求める（パターン2）
        ↓
7. 必要に応じてストーリー・play() を生成する（パターン5）
        ↓
8. CI で全テストを実行して完了
```

---

## 注意点・限界

| 注意点 | 詳細 |
|---|---|
| ハルシネーション | 存在しない API・メソッドを使ったコードが生成されることがある。必ず動作確認する |
| アンチパターン | `getByTestId` や `fireEvent` など古いパターンが生成されやすい。プロンプトで制約を明示する |
| 過剰なテスト | 似たようなテストを大量生成することがある。ケース設計を先に行って数を絞る |
| 仕様の理解不足 | コードだけ渡すと仕様の意図を誤解することがある。仕様もセットで渡す |
| セキュリティ | 本番の機密情報（API キー・個人情報）をプロンプトに含めない |
